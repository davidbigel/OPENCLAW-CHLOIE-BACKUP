# PR1 - תכנית עבודה ל-OKF_OS

מסמך עבודה לתכנון ולניהול המהלך של OKF עבור David.

## מטרה

לבנות מערכת OKF אישית, Mobile First, שמאפשרת:

- להזין מידע דרך ערוץ ה-WhatsApp הנוכחי
- לבצע שאלות על הידע שנאסף
- לקבל תשובות קצרות ומדויקות
- לתת פידבק כשיש שגיאה או חוסר
- לנהל את הידע בצורה שנשמרת, ניתנת לחיפוש, וניתנת לשיפור מתמשך

## מה חדש ב-Wiki LLM

ממחקר OKF עולה שהחידוש המרכזי הוא לא עוד "כלי", אלא פורמט:

- ידע נשמר כקבצי Markdown עם YAML frontmatter
- כל מסמך הוא יחידת ידע אחת
- קישורים רגילים בין מסמכים יוצרים גרף ידע
- index.md מאפשר גילוי הדרגתי
- log.md מאפשר היסטוריית שינויים
- הפורמט מכוון גם לבני אדם וגם לסוכנים

בפועל, זה מנסח בצורה רשמית את דפוס ה-LLM-wiki: ידע חי, גרסאי, וניתן לניווט כמו קוד.

## עקרונות מוצר

- Mobile First: כל פעולה עיקרית צריכה לעבוד טוב גם מהטלפון בלבד
- WhatsApp First: הערוץ הנוכחי הוא נקודת הכניסה הראשית
- Single User First: כרגע יש משתמש יחיד, ולכן לא בונים מורכבות של multi-tenant
- Git-Friendly: כל הידע חייב להיות נשמר כקבצים שאפשר לגבות, לבדוק, ולבקר
- Progressive Disclosure: לא מציפים הכל בבת אחת, אלא פותחים ידע בהדרגה

## Scope MVP

### In scope

- קליטת הודעות WhatsApp כקלט למערכת
- יצירת מסמכי ידע תקניים ב-OKF
- שאילתות על הידע הקיים
- מענה עם קישורים למקורות פנימיים
- מנגנון פידבק לתיקון או עדכון ידע
- יומן ריצה ומעקב גרסאות

### Out of scope כרגע

- multi-user permissions
- הרשאות ארגוניות מורכבות
- ממשק דסקטופ מלא
- חיפוש סמנטי מתקדם בכל הפלטפורמות
- אוטומציה אגרסיבית בלי בקרת איכות

## תכנית עבודה

### שלב 1: הגדרת מודל הידע

- להגדיר אילו סוגי מסמכים קיימים ב-OKF_OS
- לקבוע naming convention לקבצים
- להחליט איזה metadata חובה לכל מסמך
- להגדיר איך מסמכים מקושרים זה לזה

### שלב 2: מבנה תיקיות

- 00_raw/ לקלטים מקוריים
- 01_transcripts/ לתמלולים
- 02_requirements/ לדרישות
- 03_questions/ לשאלות פתוחות
- 04_final/ לגרסה מאושרת
- 05_runs/ ליומני ריצה
- 99_archive/ לגרסאות שהוחלפו

### שלב 3: Intake דרך WhatsApp

- כל הודעה נכנסת עוברת סיווג
- מידע חדש נשמר כ-raw בלי שינוי
- שאלות מזוהות ונשלחות למסלול טיפול
- פידבק מסומן לעדכון ידע קיים

### שלב 4: יצירת Knowledge Objects

- כל רעיון, דרישה, או החלטה נשמרים כיחידות OKF
- לכל יחידה יש title, type, description, tags, timestamp
- מסמכים מקבלים קישורים פנימיים למסמכים קשורים

### שלב 5: Query ו-Answer

- המערכת מחפשת קודם במסמכי הליבה
- אם יש חסר, היא מחזירה שאלה ממוקדת במקום לנחש
- תשובות קצרות, עם אפשרות להעמקה לפי בקשה

### שלב 6: Feedback Loop

- כשיש טעות, המערכת מסמנת גרסה חדשה
- הגרסה הישנה נשמרת בארכיון
- מתווסף run log שמתעד מה השתנה ולמה

### שלב 7: Validation

- בדיקת completeness
- בדיקת broken links
- בדיקת כפילויות
- בדיקת metadata חסר
- בדיקת התאמה ל-Mobile flow

## תוצרים נדרשים

- תכנית עבודה מאושרת
- מבנה OKF ראשוני
- Run log מסודר
- מסמכי final ל-04_final
- סט שאלות פתוחות ל-03_questions

## קריטריוני הצלחה

הפרויקט נחשב מוצלח אם:

1. אפשר להזין מידע דרך WhatsApp בלי לעצור את העבודה
2. אפשר לשאול שאלות ולקבל תשובות על בסיס הידע שנשמר
3. אפשר לתת פידבק ולקבל עדכון גרסאי מסודר
4. הידע נשאר קריא, נייד, וניתן לשיתוף
5. משתמש יחיד יכול לנהל את המערכת בלי מחשב נייח

## סיכונים

- הידע יתפזר בלי taxonomy ברורה
- יותר מדי אוטומציה תייצר רעש
- UX לא יישאר Mobile First אם נבנה סביב דסקטופ
- בלי versioning נהיה עם "זיכרון" לא אמין
