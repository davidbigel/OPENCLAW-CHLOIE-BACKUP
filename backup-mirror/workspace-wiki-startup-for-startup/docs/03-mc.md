# Mission Control Spec

Mission Control is the question-answering interface for the Startup for Startup knowledge system.

The user asks a question, chooses which knowledge sources must be used, and Snoracle returns a practical, evidence-grounded answer. The interface should make it obvious that work is happening, which sources are running, what each source found, where sources disagree, and what Snoracle thinks after reading all source answers.

## Decision State

Current agreed direction:

- Build a real v1, not a throwaway demo.
- Use a minimal Python backend plus vanilla HTML/CSS/JS.
- Public frontend runs on port 40006.
- Private agent/backend API runs on port 40007 and binds to localhost only.
- Store all mission-control state in SQLite from the start.
- Keep the public gate server-side for v1, but do not expose the literal gate value in HTML or browser JavaScript.
- Expose the public app over HTTPS at 187.124.10.241:40006.
- Always use source workers/subagents for checked sources.
- Use deep answers over fast answers; latency is not the priority.
- Ask Snoracle/source workers with the highest available reasoning effort, currently xhigh.
- Do not expose private reasoning traces in the UI; show progress and final reasoning summaries instead.
- Obsidian is not built yet, so selected Obsidian source returns "no Obsidian yet" until the vault exists.
- Start raw search with plain grep/ripgrep-style text search, not a vector index.
- No caching for now.
- Build good logging and catch/log errors throughout the flow.
- Save all answers in SQLite; export only meaningful or impactful answers as markdown snapshots.
- If raw evidence contradicts the wiki, patch the wiki with a background subagent/process. If a wiki patch process is already running, queue/wait until it finishes.

## Product Purpose

Mission Control should answer questions from:

1. WikiLLM: the curated wiki under kb/wikillm/.
2. Obsidian: the human-facing vault layer, once created.
3. Raw sources: transcripts and source metadata under kb/sources/.

The answer must distinguish:

- what the corpus says
- what raw transcript evidence says
- what Obsidian/human notes say, once available
- what Snoracle thinks
- what comes from outside knowledge
- what is uncertain or unsupported

Voice: Hebrew-first, practical truth teller, no mercy. Blunt is useful. Unsupported confidence is not.

Mission Control should refuse almost nothing. If the KB has no data about a subject, answer that clearly. If outside knowledge is used, label it clearly as outside knowledge.

## Source Checkbox Semantics

The V in a checkbox means the source must be used.

This is not "allowed if useful." It is a boolean instruction from the user:

- checked: run this source worker
- unchecked: do not run this source worker

The UI should still show different source states:

- enabled: the user checked this source
- queued: waiting for worker start
- searching: worker is retrieving material
- interpreting: worker is writing its source-specific answer
- used: evidence/answer from this source was included
- no useful evidence: source ran but found nothing helpful
- unavailable: source is selected but not implemented yet, e.g. Obsidian before the vault exists
- failed: source worker errored

If WikiLLM and Obsidian contain overlapping content, bring both. Do not pretend they are independent evidence, but do show both boxes. Snoracle should say when two boxes are duplicative.

If raw sources contradict the wiki, raw sources win. Snoracle must mention the contradiction and queue or make the needed wiki update.

Raw transcript quotes are shown even when Supadata transcription quality is imperfect. Supadata is currently the source of truth.

## Main UI

### Left Side Nav

Purpose: question history and saved research threads.

Each history item should show:

- question text, truncated
- created time
- answer status: done, running, failed, cancelled
- source badges: wiki, obsidian, raw
- optional short title generated from the question
- thread marker when the question belongs to a saved research thread

Day-one controls:

- open previous answer
- rerun previous question with the same or changed source settings
- cancel current answer
- save/unsave a research thread

Not required on day one:

- searchable history
- advanced filters

### Question Bar

Top input:

- placeholder: "Ask your question"
- supports Hebrew and English
- Enter submits
- Shift+Enter creates a newline
- submit button uses an icon plus short label

Source controls sit near the question bar:

- WikiLLM checkbox
- Obsidian checkbox
- Raw Sources checkbox

Default source setting:

- WikiLLM: on
- Obsidian: on if the user wants it, but returns "no Obsidian yet" until the vault exists
- Raw Sources: on when exact evidence matters; easy one-click toggle

### Loading State

Use CSS animation, not a GIF.

Required visible Hebrew loading text:

- "סנורקל עובד, לא לדאוג, תשובה בדרך..."

Also show the three-dot animation and every meaningful source step:

- Snoracle accepted the question
- WikiLLM worker started
- WikiLLM worker searching
- WikiLLM worker interpreting
- Raw worker started
- Raw worker found N matches
- Obsidian worker unavailable / no Obsidian yet
- Snoracle synthesizing final answer
- answer saved

The UI should show every retrieval step, not only milestones.

### Answer Area

Priority: final answer first.

Layout:

1. User question.
2. Hebrew loading/status line while running.
3. Summary answer from Snoracle, a few lines.
4. Three always-visible source boxes:
   - WikiLLM answer box
   - Obsidian answer box
   - Raw Sources answer box
5. Footer with sources used, run status, and logs/error link if needed.

Source panes are always visible. If a source was unchecked, its box says "not selected." If selected but unavailable, it says why.

The first version should be generally aesthetic, not over-designed. Clean layout, readable typography, good spacing, and no sloppy states.

## Source Boxes

### WikiLLM Box

Shows:

- source-specific answer
- wiki pages searched
- relevant concepts, claims, playbooks, questions, and episode pages
- citations already present in the wiki
- contradictions found against raw sources
- suggested wiki updates

### Obsidian Box

Until the vault exists:

- status: unavailable
- message: "אין Obsidian עדיין"

After the vault exists, shows:

- human notes searched
- linked pages
- annotations
- whether the note is original human annotation or a mirror of WikiLLM

### Raw Sources Box

Shows:

- all text-search matches
- expandable transcript quotes, default closed
- episode title when known
- video ID
- timecode
- YouTube timestamp link
- transcript file path

If raw search returns 200 matches, list all 200 in the raw box. The synthesis may summarize, but the evidence table should not silently drop matches.

## Citation Standard

Mission Control citations should be transcript/video-first and practical, not academic decoration.

Required citation fields for source-backed claims:

- episode title
- video ID
- timecode
- YouTube timestamp link
- exact transcript quote when using raw evidence
- wiki/Obsidian page path when using processed notes

Inline citation format:

- [Title | videoID | HH:MM:SS]

Expanded citation popup:

- Title
- Video ID
- Timecode
- YouTube timestamp link
- Source file/page path
- Exact quote
- Source type: WikiLLM / Obsidian / Raw / Snoracle synthesis / Outside knowledge

Every answer paragraph needs a citation or an explicit label:

- Corpus: for corpus-backed claims
- Raw: for raw transcript-backed claims
- Obsidian: for vault-backed claims
- Snoracle thinks: for synthesis
- Outside knowledge: for non-corpus knowledge
- No KB evidence: when the KB does not support the claim

Transcript quotes should be expandable by default, not fully expanded inline. The user should be able to click from a claim to a popup with exact wiki page, raw transcript location, and timestamp link.

Every answer has a "Sources Used" footer.

## Deployment And Ports

Current port check:

- port 40006 was free when checked

Port plan:

- 40006: public frontend/web app, reachable from outside
- 40007: private backend/agent API, localhost only

Firewall plan:

- allow TCP 40006 through UFW when deploying externally
- do not expose 40007
- bind 40007 explicitly to 127.0.0.1

Recommended URLs:

- public UI: https://187.124.10.241:40006/
- local UI: https://localhost:40006/
- private API: http://127.0.0.1:40007

The server-side gate is the v1 public guardrail. It is not strong production auth. The server should still exchange a valid gate entry for a short-lived HttpOnly cookie and then remove any gate parameter from the visible URL by redirecting to /.

No OpenClaw, Supadata, LLM, or Gateway token may appear in browser JavaScript.

## Browser And Request Protection

The browser should call only the public 40006 app.

Protection rules for POST requests:

- no CORS
- same-origin requests only
- allowed external origin: https://187.124.10.241:40006
- validate Origin and Referer headers when present
- require the HttpOnly mission-control auth cookie
- require a CSRF token for state-changing POST requests
- set SameSite=Lax or SameSite=Strict on the auth cookie
- set body-size limits
- log rejected requests without leaking secrets

40006 can proxy safe requests to 40007, or it can call 40007 internally server-to-server.

40007 protection:

- bind to 127.0.0.1 only
- require an internal shared secret header from the 40006 server
- reject requests from non-loopback interfaces
- never serve static browser assets
- own all filesystem, SQLite, source retrieval, and OpenClaw dispatch access

## How Questions Reach Snoracle

The browser cannot talk directly to Snoracle safely. It needs the 40006/40007 backend bridge.

Agreed flow:

1. User submits a question in Mission Control.
2. Browser sends POST /api/ask to the public 40006 app.
3. 40006 validates auth, origin, CSRF, and request shape.
4. 40006 forwards the internal job to 40007.
5. 40007 checks whether Snoracle is already busy.
6. If busy, API returns immediately with a busy response and asks the user to try again in a few minutes.
7. If available, 40007 creates a SQLite question record.
8. 40007 starts one worker per checked source.
9. Each source worker searches, interprets, and writes a source-specific answer.
10. 40007 sends the worker outputs to the persistent Snoracle session.
11. Snoracle writes the summary answer and structured final JSON.
12. 40007 stores the final answer, source answers, evidence, logs, and errors in SQLite.
13. If raw evidence contradicts the wiki, 40007 starts or queues a background wiki-patch subagent/process.
14. If the answer is meaningful or impactful, 40007 exports a markdown snapshot.
15. 40006 returns or streams the result to the browser.

Snoracle is persistent, not a fresh generic LLM call per question.

Recommended persistent session identity:

- session key: session:mission-control-snoracle
- store this in SQLite settings table as snoracle_session_key
- also keep it configurable through env var MC_SNORACLE_SESSION_KEY

Dispatch adapter:

- implement one internal function: dispatch_to_snoracle(question_id, evidence_pack)
- the adapter is the only place that talks to OpenClaw/Gateway/session APIs
- source retrieval and browser code must not know OpenClaw internals

If OpenClaw dispatch changes later, only the adapter changes.

## Source Workers / Subagents

Source workers run from v1 for every checked source.

Reason: avoid context poisoning and keep each source's interpretation isolated.

Worker rules:

- workers retrieve evidence
- workers interpret their own source
- workers write a source-specific answer
- workers return structured JSON
- workers must include citations or explicit "no evidence" labels
- workers must log errors
- workers do not write the final cross-source answer

Snoracle rules:

- receives all worker outputs
- compares source answers
- shows disagreements
- treats raw sources as highest authority when raw contradicts processed wiki pages
- writes the summary answer above the three boxes
- returns final JSON for the UI
- starts or queues wiki updates when raw contradictions or real new insights appear

## API Sketch

### POST /api/ask

Public route on 40006.

Request:

~~~json
{
  "question": "מה הקורפוס אומר על GTM לסטארטאפים בתחילת הדרך?",
  "sources": {
    "wikillm": true,
    "obsidian": true,
    "raw": true
  },
  "mode": "deep_answer"
}
~~~

Immediate response when accepted:

~~~json
{
  "question_id": "q_20260630_001",
  "status": "running"
}
~~~

Immediate response when Snoracle is busy:

~~~json
{
  "status": "busy",
  "message": "Snoracle is busy with another question. Try again in a few minutes."
}
~~~

### GET /api/questions/:id

Public route on 40006.

Response:

~~~json
{
  "question_id": "q_20260630_001",
  "status": "done",
  "question": "...",
  "sources_selected": ["wikillm", "obsidian", "raw"],
  "sources_used": ["wikillm", "raw"],
  "summary_answer_markdown": "...",
  "boxes": {},
  "sources_used_footer": "..."
}
~~~

### POST /api/questions/:id/cancel

Cancels a running question if possible.

If a worker cannot be killed cleanly, mark the question as cancelling, ignore late worker results, and log the late completion.

### POST /api/questions/:id/rerun

Reruns a previous question with the same or changed source settings.

### GET /api/questions/:id/events

Streaming route for status updates using Server-Sent Events.

The UI should update source boxes and progress text from these events.

### evidence_items

Fields:

- rank

### wiki_update_candidates

Use when a question reveals:

- a real new insight
- a contradiction between raw source and wiki
- a weak or missing wiki page
- a useful answer worth digesting into kb/wikillm/questions/

Be picky. Do not fill the wiki with low-value one-off questions.

### answer_snapshots

SQLite is the source of truth for all answers. Markdown snapshots are only for meaningful or impactful answers worth preserving outside the database.

Suggested export path:

- kb/mission-control/answers/<question_id>.md

Export when:

- the answer contains a real new insight
- the answer is likely to be reused
- the answer explains an important contradiction
- the answer is useful as a research artifact
- David explicitly asks to preserve it

### wiki_patch_jobs

Use for background wiki updates after Mission Control finds a contradiction or meaningful improvement.

Rules:

- raw-vs-wiki contradiction creates a patch job
- one wiki patch process runs at a time
- if a patch process is already running, the new job waits in queue
- patch worker/subagent edits kb/wikillm/ directly
- patch worker logs changed files and evidence used
- Mission Control marks the job done, failed, or waiting

## Retrieval Strategy

### WikiLLM

Search:

- kb/wikillm/index.md
- kb/wikillm/claims/
- kb/wikillm/concepts/
- kb/wikillm/playbooks/
- kb/wikillm/questions/
- kb/wikillm/episodes/

Return:

- page title
- path
- relevant excerpt
- citations already present in page
- confidence if available
- source-specific interpretation

### Obsidian

Current v1 behavior:

- if selected, return "אין Obsidian עדיין"
- status: unavailable

Future behavior:

- search human notes
- search linked pages
- identify annotations
- label whether a note is original human annotation or a wiki mirror

### Raw Sources

Search:

- kb/sources/raw/<videoID>.md
- kb/sources/lists/

Method:

- use grep/ripgrep-style text search first
- return all matches
- store all matches in SQLite
- show all matches in the raw source box

Return:

- exact quote
- video ID
- title if known
- timecode
- YouTube timestamp link
- transcript path

Raw source mode is mandatory when checked. Raw source evidence overrides wiki when they conflict.

## Logging And Errors

Everything should be wrapped with explicit error handling and logging.

Log:

- request accepted/rejected
- auth failures
- source worker start/end
- search commands and match counts
- Snoracle dispatch start/end
- JSON parse failures
- SQLite errors
- cancellation events
- wiki update candidate creation
- answer snapshot export
- wiki patch job queued/started/finished

Suggested log paths:

- kb/mission-control/logs/app.log
- kb/mission-control/logs/questions/<question_id>.log

UI behavior on errors:

- show a clear failed state in the relevant box
- keep successful source boxes visible
- preserve logs for debugging
- never expose secrets in the browser

## Answer Rules

Snoracle must:

- answer in Hebrew unless the user asks otherwise
- preserve English startup terms when useful
- cite every paragraph or label it clearly
- label corpus evidence, raw evidence, Snoracle synthesis, and outside knowledge
- say when evidence is weak or missing
- avoid generic startup advice unless labeled as outside knowledge or synthesis
- challenge the user's assumptions when needed
- be blunt when an idea is unsupported, generic, or contradicted
- show disagreements between source workers
- patch or queue wiki updates when raw evidence contradicts the wiki

Bad answer:

- "Startups should always focus on product-market fit."

Better answer:

- "Corpus: הקורפוס תומך בזה רק חלקית: בפרקים X ו-Y יש דגש על PMF, אבל בפרק Z הדגש הוא דווקא על הפצה מוקדמת. [Title | videoID | 00:12:34]"
- "Snoracle thinks: ההמלצה המעשית היא לא להפוך PMF לסיסמה. אם אין הפצה מוקדמת, גם מוצר טוב עלול להיתקע."
