# Mission Control Immediate Change Plan

Date: 2026-07-08

## Update - Implemented on 2026-07-08

- The frontend no longer polls question status. Live asks now wait on one blocking `POST /api/ask` and reruns use the same pattern.
- Cancel for an in-flight blocking ask now works through a temporary `client_request_id` mapping instead of waiting for a returned `question_id`.
- The default OpenClaw runtime budget remains `600s`.
- The loader counter remains in the status bar and shows elapsed time in the shape `17/600s(max)`.
- To reduce hangs, the UI no longer preselects `obsidian`, the OpenClaw turn now uses `--thinking high` instead of `xhigh`, and the Mission Control prompt explicitly asks for fast, non-exhaustive retrieval.
- OpenClaw timeout wrappers are now classified as real Mission Control failures instead of surfacing as misleading raw-success responses.
- Live verification after implementation: question `q_20260708_125143_438920` completed through the real OpenClaw path in about 56 seconds with a final direct `POST /api/ask` response, `parse_status=parsed`, and run id `81027f73-f53b-4f26-a440-78cc9b345b5e`.

This plan replaces the broader next-step draft. For now, David asked for only two changes:

1. Raise the Mission Control OpenClaw runtime budget to `600s`.
2. Add a visible JS-only elapsed counter in the loader, in the shape `112/600s(max)`.

No other behavioral changes are in scope for this pass.

## Current decision

The app will keep the existing request model for now:

- frontend still submits one question and polls status
- backend still runs the OpenClaw turn in a background thread
- no synchronous `POST /api/ask` conversion in this pass
- no source-default changes in this pass
- no prompt/agent-strategy changes in this pass

The goal is narrower:

- give the user a longer runtime budget before Mission Control times out
- make the waiting state visibly alive so the UI does not look frozen

## Change 1 - Runtime budget to 600 seconds

### What changes

- change the default `MC_SNORACLE_TIMEOUT_SECONDS` from `300` to `600`
- keep the same OpenClaw dispatch path, just with a larger timeout budget
- update the timeout error string so it reports the real configured limit
- update tests and docs that still assume `300`

### Why

The live stuck run on `q_20260708_112732_210006` consumed about `302s` and timed out. David asked to extend the runtime instead of changing the retrieval or request model right now.

### Non-goals

- no prompt tightening
- no lower reasoning mode
- no timeout-wrapper redesign

## Change 2 - JS-only loader counter

### What changes

- keep the current loader and polling flow
- add a client-side seconds counter while a question is queued or running
- display it in the status bar as `<elapsed>/<max>s(max)`
- use `600` as the visible max to match the new runtime budget

### Target behavior

- the counter starts immediately when the user submits a question
- if the page is rendering a currently running question, the counter derives from the saved question timestamps when possible
- the counter stops when the question reaches `done`, `failed`, or `cancelled`
- the counter is pure browser-side UI and does not require backend schema changes

### Why

The current UI does poll, but from the user's perspective the page looks stagnant. A visible second counter makes it obvious that Mission Control is still active.

## Acceptance criteria

- OpenClaw command construction uses `--timeout 600` by default.
- A running question shows a visible live counter such as `17/600s(max)`.
- The counter stops once the question finishes or is cancelled.
- Mission Control is restarted and serving after the change.

## Deferred work

These ideas are intentionally deferred in this pass:

- removing polling
- converting `POST /api/ask` into a blocking final-response request
- changing source defaults
- lowering reasoning budget
- changing the agent prompt contract
- redesigning timeout handling beyond the longer budget
